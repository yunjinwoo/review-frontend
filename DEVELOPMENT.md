# 프론트엔드 개발 정리 (DEVELOPMENT.md)

이 문서는 **AI Code Reviewer** 프로젝트의 프론트엔드 영역에서 집중적으로 개발된 기능과 아키텍처 구조의 변화, 그리고 직면했던 이슈들의 해결책을 기록한 프론트엔드용 문서입니다.

---

## 1. 🚀 핵심 컴포넌트 개발 내역 (Features)

### 1) 폴더/파일 무제한 드래그 앤 드롭 (Drag & Drop) 기능 구현
* **의도**: 코드 리뷰 속도를 높이기 위해, 파일을 일일이 복사+붙여넣기 하는 과정에서 벗어나 파일 및 폴더 전체를 드래그하여 코드를 자동 첨부하기 위함입니다.
* **구현 방식**:
  - `onDragOver`, `onDragLeave`, `onDrop` 이벤트 핸들러를 텍스트박스 부모 컨테이너에 바인딩하고 드래깅될 때 Dim 처리 및 애니메이션 피드백을 추가했습니다.
  - `e.dataTransfer.items`의 `webkitGetAsEntry()` API를 사용하여 깊스(Depth)가 있는 폴더가 드롭되더라도 하위 항목을 재귀적으로 완벽히 읽어오도록 구성했습니다.
* **성능 제어 규칙**:
  - 시스템 브라우저 크래시를 방지하고자 읽는 파일 개수를 **최대 10개**, 단일 파일당 용량을 **2MB**로 제한했습니다.
  - 정규식을 활용해 빌드 폴더(`node_modules`, `.next`, `dist` 등)와 이미 지나치게 큰 바이너리 파일(`.jpg`, `.mp4`, `.pdf` 등)은 제외시켜 순수 코드만 가져올 수 있게 최적화했습니다.

### 2) 3단 레이아웃 (3-Column Grid) 개편 및 히스토리 데이터 연동
* **의도**: 일회성 질의응답을 넘어 사용자의 과거 리뷰 내역을 추적하고, 과거 코드를 즉시 복원해 주기 위함입니다.
* **구현 방식**:
  - 기존 2분할(입력/결과 양방향) 화면에 `Tailwind CSS`의 Grid 시스템(`grid-cols-12`)을 사용하여 좌측 패널(Col 1-2)에 히스토리 리스트 구조를 짰습니다.
  - 컴포넌트가 처음 마운트 될 때 `useEffect`를 통해 백엔드(GET API) 정보를 로딩(`loadHistory()`)해 리스트에 그리도록 추가했습니다.
  - 패널 내의 각 레코드 버튼을 누르면 해당 리뷰 ID(`loadReviewDetails()`)를 다시 Fetch하여 좌측/우측 내용을 과거 데이터 상태로 덮어쓰도록 복원 로직을 작성했습니다.

### 3) 사용자 경험(UX) 개선 및 디자인 고도화
* 각 다른 파일들이 텍스트 입력창 한곳으로 병합될 때, 리뷰어가 컴포넌트 맥락을 명확히 이해하도록 `// --- File: {path} ---` 와 같은 경로 포맷을 자동 생성해주었습니다.
* 파일과 리뷰가 쌓인 화면을 클린하게 초기화할 수 있는 **초기화 버튼** 상태 관리를 도입했습니다.

## 2. 🐛 트러블슈팅 및 예외 대응 (Troubleshooting)

### 1) 프론트엔드 백화현상 및 무한 로딩 오류
* **현상/원인**: 백엔드의 AI 리뷰 API에서 모델 쿼터 초과 문제로 500 에러를 반환했을 때, 예외 처리는 되어있었으나 로딩 아이콘(`setIsLoading(false)`)이 풀리지 않거나 에러 내용을 노출해주지 않아(흰 빈 화면) 후속 조작을 할 수 없는 잠금 현상이 나타났습니다.
* **해결 방법**: 
  - `try-catch` 문의 `finally` 영역을 보강하여 무조건 `setIsLoading(false)`가 호출되도록 정리했습니다.
  - 서버 측 API가 에러 상황 자체를 마크다운 형태(`data.review = error msg`)로 던져주게 설계를 합의하였기 때문에 프론트엔드는 실패의 경우에도 해당 마크다운을 결과창에 친절하게 띄우도록 수정하여 UX 혼란을 막았습니다.
  - 더불어 실패(에러)가 반환됐어도 백엔드가 로그 기록 차원에서 남겼을 수 있으므로 항상 `finally` 과정에서 `await loadHistory();` 를 호출하여 히스토리 사이드바를 리렌더링시키도록 처리했습니다.

### 2) File Reader 재귀 비동기 처리와 상태 업데이트 꼬임 현상
* **현상/원인**: 복수의 파일과 폴더트리를 파싱할 때 `FileReader.readAsText()` 동작이 비동기적으로 중구난방 실행되어 텍스트 결과 배열과 React의 상태(State)가 충돌할 우려가 매우 컸습니다.
* **해결 방법**:
  - `Promise` 체이닝 및 `async/await` 패턴이 적용된 별도의 `getFilesFromEntry()` 유틸리티 함수를 격리시켜 만들고 제한 객체 컨텍스트(`resultContext.files`)를 공유하게 만들었습니다. 
  - 모든 파일 파싱이 (최대 제한수를 채우거나, 전체 탐색이) 완벽하게 끝난 뒤에만 `Promise.all` 처럼 처리해 `setCode()`를 단 1번 일어나게 통제하여 리렌더링 부하 및 Race Condition 문제를 해결했습니다.
